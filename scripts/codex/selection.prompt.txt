== THE BUG (precise repro, already characterized) ==
`setLocalSelection(anchorX, anchorY, focusX, focusY, bg, fg)` does NOT shade the local cell grid we
expect. A FIXED probe of `setSelectionRange(0,0,5,0)` (select the first 5 columns of the first
visible line) shades `y=5, x=28..46` in period-4 groups (3 shaded cells + 1 gap, repeating) — i.e. a
~4× scale + offset in BOTH axes. A real selection on document line N lands roughly `5 + 4*N` rows too
low. So `y = 5 + 4*localY` (0→5, 3→17 both observed) and x is similarly scaled. The TEXT itself
renders correctly (greeter.ts content at rows y=1..8), so the render buffer is right — only the
selection mapping is wrong.

Hypotheses to test (reduce, don't assume): (1) `setLocalSelection` coords are relative to the
renderable's own viewport and require `textBufferView.setViewport(x,y,w,h)` / `setViewportSize(w,h)`
to be set to the code renderable's laid-out size first — which may not have happened when
`applySelection` runs (yoga layout may lag the content set). (2) coords may be logical-line based
while we render only the visible window (so anchorY should be absolute doc-line, with the view
scrolled, not window-relative). (3) the ~4 factor may be a supersample/scroll artifact — inspect
`_scrollY`/`updateViewportOffset`. Confirm the true contract from the source, then fix `applySelection`
(and/or `SelectableText`) to pass coords in whatever space OpenTUI actually wants. If native selection
genuinely cannot be driven programmatically to the right cells, document why with evidence and
propose the minimal alternative — but exhaust the native path first.

  H=scripts/tui-harness.sh; S=sel-$$; BUN="$HOME/.bun/bin/bun"
  "$H" launch "$S" 120x40 env TUI_FRAME_DUMP=1 TUI_SEL_RENDER=1 bun run src/main.ts "$PWD/fixtures" >/dev/null

  "$H" send "$S" Right >/dev/null; "$H" send "$S" Down >/dev/null; "$H" send "$S" Down >/dev/null   # focus editor, move to a content line
  "$H" settle "$S" >/dev/null 2>&1; sleep 0.4; cp artifacts/frame.json /tmp/A.json
  for _ in 1 2 3; do "$H" send "$S" S-Right >/dev/null; done                                        # select 3 graphemes on the SAME line
  "$H" settle "$S" >/dev/null 2>&1; sleep 0.4; cp artifacts/frame.json /tmp/B.json
  # cursor line & selection are in artifacts/status.json (fields: cursor{line,col}, selection{start,end})
  # DIFF /tmp/A.json vs /tmp/B.json: the bg-changed cells are the highlight. PASS iff they land on the

  "$H" kill "$S" >/dev/null 2>&1

frame.json shape: { width, height, rows:[{ y, text, bg:[...], fg:[...], attrs:[...] }] }. Cell (x,y)
bg = rows[y].bg[x]. The selection bg differs from the surrounding bg only on selected cells. Map the
cursor's doc line to its buffer row: editor content starts at buffer y=1 (editor top border), so doc
line L at scrollTop T renders at y = 1 + (L - T). The gutter occupies the left columns of the editor
area; the CODE renderable starts after it — account for that when checking the x columns.
